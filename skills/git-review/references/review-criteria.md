# Review Criteria - 代码审查标准

## 评分维度

### 1. 功能正确性与健壮性（40分）

检查项：
- [ ] 逻辑是否正确，是否满足需求
- [ ] 边界情况处理（空值、零值、极端输入）
- [ ] 异常处理是否完善
- [ ] 错误信息是否清晰有用
- [ ] 是否有明显的 bug

扣分示例：
- 未处理 null/undefined：-5分
- 缺少边界检查：-3分
- 异常被静默吞掉：-5分
- 逻辑错误：-10分

### 2. 安全性与潜在风险（30分）

检查项：
- [ ] SQL 注入风险
- [ ] XSS 跨站脚本攻击
- [ ] 敏感信息泄露（密码、密钥、token）
- [ ] 输入验证是否充分
- [ ] 权限检查是否到位
- [ ] CSRF 防护

扣分示例：
- SQL 拼接：-15分
- 用户输入未转义：-10分
- 硬编码密钥：-20分
- 缺少权限校验：-10分

### 3. 最佳实践（20分）

检查项：
- [ ] 命名规范（变量、函数、类）
- [ ] 代码结构清晰
- [ ] 单一职责原则
- [ ] DRY 原则（避免重复）
- [ ] 注释是否必要且清晰
- [ ] 代码风格一致性

扣分示例：
- 命名不清晰：-3分
- 函数过长（>50行）：-5分
- 重复代码：-5分
- 深层嵌套（>4层）：-3分

### 4. 性能与资源利用（5分）

检查项：
- [ ] 是否有不必要的循环
- [ ] 数据库查询是否高效
- [ ] 内存使用是否合理
- [ ] 是否有资源泄漏风险

扣分示例：
- N+1 查询问题：-3分
- 不必要的大对象创建：-2分
- 未关闭的资源连接：-3分

### 5. Commits信息的清晰性与准确性（5分）

检查项：
- [ ] 提交信息是否清晰描述了变更内容
- [ ] 提交信息是否准确反映了实际修改
- [ ] 是否便于后续维护和协作
- [ ] 是否遵循团队的 commit 规范（如 Conventional Commits）

扣分示例：
- 提交信息过于模糊（如"fix"、"update"）：-2分
- 提交信息与实际修改不符：-3分
- 多个不相关的修改合并在一个提交：-3分

## 问题严重程度

| 级别 | 图标 | 说明 | 示例 |
|------|------|------|------|
| 严重 | 🔴 | 必须修复，会导致 bug 或安全问题 | SQL注入、空指针、逻辑错误 |
| 中等 | 🟡 | 应该修复，影响代码质量 | 命名不清、重复代码、缺少错误处理 |
| 建议 | 🟢 | 可选优化，提升代码质量 | 性能优化、注释补充、代码简化 |

## 常见问题检查清单

### Python
- [ ] 异常处理是否使用具体异常类型
- [ ] 是否使用了 f-string 格式化
- [ ] 是否有类型注解
- [ ] 是否遵循 PEP 8

### JavaScript/TypeScript
- [ ] 是否使用 const/let 而非 var
- [ ] 异步操作是否正确处理
- [ ] 是否有类型定义（TS）
- [ ] 是否避免了 any 类型

### 通用
- [ ] 是否有 TODO/FIXME 未处理
- [ ] 是否有调试代码（console.log、print）
- [ ] 是否有硬编码的配置值
